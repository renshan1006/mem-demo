# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**DataGuardian** — a multi-agent data governance platform for Master Data Management (MDM). It matches and deduplicates customer/product entities across heterogeneous sources (CRM, ERP, ECommerce), using **bge-large-zh embeddings + a weighted rule engine + Dify LLM**, then routes borderline matches to a human review UI. A newer agent layer is being built on top of the original matching pipeline (see Architecture below).

The whole pipeline runs on **synthetic data** generated in code (`scripts/generate_dataset.py`, ~6,200 rows, fixed `seed=42`); no real PII is involved.

## Common Commands

All `scripts/*.py` resolve paths via `BASE_DIR = Path(__file__).resolve().parent.parent`, so they run from anywhere — but several do flat imports of sibling modules (`from embedder import ...`), so **run pipeline scripts from inside `scripts/`** (`cd scripts` then `python <name>.py`), matching the README.

```bash
# Install (Python 3.9+)
pip install -r requirements.txt

# 1. Generate the synthetic dataset (5 CSVs into data/)
cd scripts && python generate_dataset.py

# 2. Run the matching pipelines (each writes several files into data/)
python mdm_pipeline.py            # customer matching → candidates, review_queue, golden
python product_pipeline.py        # product matching
python finalize.py customer       # UnionFind-cluster auto_merge + human-reviewed pairs → final_golden_customers.csv
python finalize.py product

# 3. Tune thresholds from review log + ground-truth labels, then re-run step 2
python tune_thresholds.py

# 4. Dify LLM enrichment (needs dify/config.json)
python dify_client.py             # connectivity self-test
python dify_enrich.py customer 20 # enrich first 20 review-queue rows

# 5. Migrate CSV outputs to a DB
python migrate_to_db.py           # SQLite (default, zero config)
python migrate_to_db.py --pg      # PostgreSQL (needs sql/pg_config.json)

# Launch the review UI (see gotcha below)
python run_streamlit.py run streamlit_app.py   # NOT `streamlit run`
```

Each agent module also has a `__main__` self-test runnable directly, e.g. `python match_agent.py`, `python evaluator.py`.

There is no test suite, linter, or build step configured. "Run a single test" maps to running an individual script or agent `__main__`.

## Critical Gotchas

- **Streamlit SSL on Windows/Py3.9**: do **not** run `streamlit run streamlit_app.py`. `run_streamlit.py` monkey-patches `ssl.SSLContext.load_default_certs` / `ssl.create_default_context` to tolerate a corrupt Windows cert store and fall back to `certifi`. Always launch through it.
- **First pipeline run is slow** (~minutes): `mdm_pipeline.py` auto-downloads `BAAI/bge-large-zh-v1.5` (~1.3 GB) into `.model_cache/`. `embedder.py` forces the `HF_ENDPOINT=https://hf-mirror.com` mirror by default (no VPN needed).
- **Graceful embedding fallback**: if `sentence-transformers`/the model can't load, the system silently falls back to `difflib` edit-distance matching. Functionally everything still runs, but the review queue balloons (~2,000 rows vs ~240). A successful load prints `模型加载完成` and `Embedding 完成`. Success/failure is the #1 cause of "matching quality is bad."
- **`accelerate` shim** in `embedder.py`: transformers ≥4.40 calls `accelerate.init_empty_weights`; if `accelerate` isn't installed, a no-op builtin shim is registered so model loading doesn't crash. Don't remove it.
- **Most `data/*` outputs are gitignored** (candidates, review queues, golden records, metrics, tuned thresholds). They must be regenerated by running the pipeline in order. Only the source CSVs are checked in (and even those are gitignored once `generate_dataset.py` regenerates them).

## Architecture

The project has **two coexisting layers** — the original linear pipeline (README-documented) and a newer agent framework being built over it (`docs/团队协作开发指南.md` is the source of truth for the agent design).

### Layer 1 — Matching pipeline (existing, stable)

```
generate_dataset.py  →  mdm_pipeline.py / product_pipeline.py  →  finalize.py  →  streamlit_app.py
                           (block → embed → score → decide)        (UnionFind)      (human review)
                                  ↑                                        │
                                  │                                        ▼
                            tune_thresholds.py  ←── review_log.csv (human decisions + is_true_match)
```

Each pipeline stage:
1. **Normalize** text (light normalization for embedding; heavier stripping for edit-distance).
2. **Block** into buckets by region/category so only same-bucket pairs are scored (avoids O(n²)).
3. **Precompute embeddings** once per unique name (`embedder.embed_dataframe`), then **score** each candidate pair. Scoring formulas:
   - Customer: `name×0.45 + tax_id×0.30 + phone×0.15 + address×0.10` (+ tax/phone bonuses)
   - Product: `name×0.20 + brand×0.15 + model×0.10 + sku×0.25 + upc×0.15 + spec×0.05` (− conflict penalties)
   - Embedding similarity is blended with edit-distance (`EMBED_BLEND = 0.80`).
4. **Categorize** by thresholds from `data/optimized_thresholds.json` → `auto_merge` / `review` / `no_match`.
5. **finalize.py** runs `UnionFind` over (auto-merge pairs ∪ human-approved pairs) to build final golden records, picking the most complete record per cluster.

`tune_thresholds.py` scans `[0.30, 0.95]` at 0.01 step, optimizing for `auto_merge` precision ≥ 0.95 while minimizing the review queue, then writes `optimized_thresholds.json`. Re-running the pipelines picks up the new thresholds — this is the "feedback retraining" loop.

### Layer 2 — Agent framework (DataGuardian, in progress)

New work lives in feature branches (`feature/agent-registry`, `feature/quality-agent`, `feature/normalize-agent`, `feature/chat-agent`). The framework:

- **`base_agent.py`** — defines the interface contract: `AgentTask` / `AgentResult` dataclasses and the `BaseAgent` ABC. Every agent must implement `name`, `description`, and `run(task) -> AgentResult`, and may override `health_check()` / `get_actions()`. Helpers `_ok()`, `_fail()`, `_timed_run()` build/times results.
- **`agent_registry.py`** — a module-level singleton `registry`. Agents register themselves; everything dispatches through `registry.call(name, task)` / `registry.call_safe(...)`. The registry logs call history and can health-check all agents. **Agents must call other agents only via the registry — never by direct import** (this is the stated inter-agent rule).
- **`match_agent.py`** (`EntityMatchAgent`) — the reference example: it wraps the existing pipeline functions (`mdm_pipeline.load_customers`, `precompute_name_embeddings`, `build_candidate_pairs`, …) into agent actions (`match_customers`, `match_products`, `metrics`, `run_dify`, `status`). Use it as the template when adding a new agent.
- **`quality_agent.py`** (`DataQualityAgent`) — M2 数据质量评估：`scan` 扫描 5 维 Baseline 检测（缺失/重复/格式/异常值/一致性），`report` 生成中文 JSON+Markdown 报告，`compare` 跨源对比（按 `canonical_*_id` 分组发现字段冲突）。0–100 加权评分（完整 30% / 唯一 20% / 有效 25% / 一致 25%），LLM 增强可选（`quality_api_key`，未配置自动降级）。CLI: `python quality_agent.py --compare --source a.csv --source b.csv`。
- **`evaluator.py`** — unified evaluation (`Precision/Recall/F1/Accuracy`) to compare Baseline vs. agent methods. Every module must report its metrics through it.

Per the team guide, four modules are planned: M0/M1 registry+match (done), M2 `quality_agent` (data quality), M3 `normalize_agent` (standardization), M4 `chat_agent` (NL → registry dispatch). When implementing these, follow the `BaseAgent` contract and the action lists specified in `docs/团队协作开发指南.md`.

### Module import convention

Pipeline scripts use flat, sibling-relative imports (`from embedder import get_embedder`). Agent modules support **both** package-style and direct-run imports via:
```python
try:
    from .base_agent import BaseAgent, AgentTask, AgentResult
except ImportError:
    from base_agent import BaseAgent, AgentTask, AgentResult
```
…and `match_agent.py` injects `scripts/` onto `sys.path` to make flat imports of the pipeline modules work. Preserve this dual pattern in new `scripts/` modules.

## Configuration & External Services

- **Dify** (`dify/config.json`, gitignored): `{ "api_key": "app-…", "base_url": "https://api.dify.ai" }`. Note `base_url` is `api.dify.ai`, **not** `cloud.dify.ai`. The key can also come from env `DIFY_API_KEY` (see `_env_or_config` in `dify_client.py`). Workflow DSL is in `dify/mdm_workflow.yml`. All Dify calls degrade gracefully (return `decision=review`) when unavailable — mirror this try/except pattern in any new LLM call.
- **Database** (`scripts/db_manager.py` `DBManager`): SQLite by default (`data/mdm.db`, gitignored), optional PostgreSQL via `sql/pg_config.json`. Access through the `get_db()` singleton.
- **Cache** (`scripts/cache_manager.py`): local-file in dev, optional Redis in prod (`sql/redis_config.json`).

## 工作约定 (Workflow Conventions)

以下约定在本仓库中**必须遵守**：

- **中文回复**：与用户交流时一律使用中文输出。
- **记录问题与解决方案**：开发中遇到报错、踩坑、非显而易见的问题时，把「问题现象 + 根因 + 解决方案」追加写入 `docs/调试报告.md`，再继续后续工作。不要把排错过程只留在对话里。
- **文档同步**：每完成一个功能或修改行为后，同步更新相关文档 —— `docs/` 下对应文档、`README.md`，以及本 `CLAUDE.md`（仅当改动影响架构/命令/约定时才需要，否则可不动）。代码与文档不得脱节。
- **即时提交**：完成一项功能或修复后立即 `git commit`（按上述提交信息格式），不要囤积改动或等待用户提醒。遵循默认原则：仅在被要求时才 `push`；提交前若处于 `master` 分支，先切到功能分支。
- **子智能体驱动开发**：优先用子智能体（subagent）驱动开发。把可独立的子任务（跨多文件的搜索/调研、并行实现的模块、代码审查、验证等）委派给子智能体执行，主线程负责拆解任务、汇总结果与决策。适合委派：大面积探索性搜索、互不依赖的并行编码、独立的审查/验证；不适合委派：需要频繁交互决策的小步改动、单行修复。
