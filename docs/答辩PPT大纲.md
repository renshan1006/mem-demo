## 第 2 页 · 问题场景

**痛点**：企业 CRM、ERP、电商系统中存储的客户和商品数据互相孤立，存在大量重复和不一致。

| 系统 A (CRM) | 系统 B (ERP) | 问题 |
|-------------|-------------|------|
| 北京阿尔法科技有限公司 | 北京 Alpha 科技公司 | 中英文混用 |
| 电话 010-1234-5678 | 电话 01012345678 | 格式不一致 |
| 海淀区中关村大街 1 号 | 海淀区中关村大街 1 号楼 | 地址后缀差异 |
| SKU427161 / 512GB | SKU518116 / 512GB | 同名不同 SKU |

**业务影响**：客户画像不准确、库存报表偏差、营销重复触达、BI 分析失真。

> 💡 **核心问题**：如何从多源异构数据中自动识别"谁是谁"，并高效完成人工复核？

## 第 5 页 · 数据集

| 数据源 | 记录数 | 实体类型 | 字段数 |
|--------|:----:|:------:|:----:|
| CRM 客户 | 1,600 | 客户 | 12 |
| ERP 客户 | 1,400 | 客户 | 12 |
| ECommerce 客户 | 1,200 | 客户 | 12 |

- **重复率**：客户 ~20%（840 对跨源重复），商品 ~20%（400 对跨源重复）

- **真值标签**：`canonical_customer_id` / `canonical_product_id` 用于评估

## 第 6 页 · 匹配算法（核心页）

### 6.1 四阶段流水线

```
文本规范化 → Embedding 预计算 → 阻塞分桶 → 多策略打分 → 阈值决策
```

### 6.2 客户匹配公式

```
Score = 名称_Embedding × 0.45 + 税号_精确 × 0.30
      + 电话_精确 × 0.15 + 地址_编辑距离 × 0.10
      + 税号加分(0.10) + 电话加分(0.05)

名称相似度 = bge-large-zh 余弦 × 0.80 + difflib 编辑距离 × 0.20
```

### 6.3 商品匹配公式（含 SKU 惩罚）

```
Score = 名称_E × 0.20 + 品牌_E × 0.15 + 型号_E × 0.10
      + SKU_精确 × 0.25 + UPC_精确 × 0.15 + 规格_编辑 × 0.05

⚠️ SKU 惩罚: 两边 SKU 都非空且不同 → 分数上限 0.55
⚠️ UPC 惩罚: 两边 UPC 都非空且不同 → 分数上限 0.55
```

### 6.4 决策策略

| 区间 | 客户 | 商品 | 动作 |
|------|:--:|:--:|------|
| auto_merge | ≥ 0.53 | ≥ 0.54 | 自动合并，无需人工 |
| review | 0.50–0.53 | 0.51–0.54 | 人工审核 + Dify LLM 辅助 |
| no_match | < 0.50 | < 0.51 | 自动判定为不同实体 |

> 🎯 **亮点 2**：商品 SKU 冲突惩罚机制——同名不同 SKU 的商品被正确识别为不同变体，避免过度合并。

## 第 7 页 · Embedding 语义匹配效果

### 为什么需要 Embedding？

| 场景 | difflib 编辑距离 | bge-large-zh 语义向量 |
|------|:--------------:|:-------------------:|
| "北京Alpha科技" vs "北京阿尔法科技" | 0.42（低） | **0.91**（高）✅ |
| "华为技术有限公司" vs "华为" | 0.36（低） | **0.78**（中）✅ |
| "上海XX贸易" vs "深圳XX贸易" | 0.89（高）❌ | **0.52**（低）✅ |

- **模型**：BAAI/bge-large-zh-v1.5，中文 C-MTEB 榜单 Top 3
- **向量维度**：1024，归一化后余弦相似度
- **混合策略**：Embedding × 0.8 + difflib × 0.2（平滑生僻词）

> 🎯 **亮点 3**：Embedding 让审核队列真匹配率从 8.8% → 39.9%，提升 4.5 倍。

## 第 8 页 · Dify 智能体（亮点页）

### 工作流拓扑

```
Start (输入) → LLM Node (DeepSeek-Chat) → Code Node (解析) → End (输出)
 30 个变量        System + User Prompt      括号计数法提取JSON    {decision,
                  Jinja2 条件分支                              confidence,
                  T = 0.1                                     reasoning}
```

### Dify 在本系统中的角色

- **定位**：模糊匹配对的"二审法官"，不是替代规则引擎
- **触发**：只对审核队列中的 272 条模糊对调用（不是全量 70 万对）
- **成本**：DeepSeek ¥0.001/1K tokens，审核队列全量约 ¥0.3

### Dify 增强效果（实测）

| | Embedding 管道判定 | Dify LLM 二次判断 |
|------|:--:|:--:|
| 商品 31 条 | 全入 review | **29 条 no_match**，仅 2 条保留 |
| 客户 10 条采样 | 全入 review | 3 条 no_match，7 条保留 |

> 🎯 **亮点 4**：规则 + Embedding + Dify LLM 三级漏斗——粗筛(规则) → 精筛(Embedding) → 终审(LLM)，每层过滤量递增。

### Dify 与前端联动

```
dify_enrich.py 调用 Dify API → 写入 review_enriched.csv
         ↓
Streamlit 详情审核页自动加载 → 展示 LLM 建议卡片
         ↓
  人工参考 LLM 建议 → 做最终决策 → 写入 review_log.csv
```

- 对审核队列前 20 条运行 `dify_enrich.py customer 20`，Dify 判了 18 条 no_match、2 条 review
- Streamlit 详情审核页左侧评分仪表下方动态展示 Dify 建议（绿/红色边框 + 置信度 + 推理）

## 第 10 页 · 阈值反馈回训

### 闭环机制

```
审核日志 + 真值标签 → tune_thresholds.py → 扫描 4,356 种组合
  → 优化目标: auto_merge 精确率 ≥ 95% + 最小化 review 队列
  → 输出 optimized_thresholds.json → 管道自动读取
  → 审核更多 → 阈值更准 → 审核更少
```

### 阈值演进

| | 默认 | 优化后(客户) | 优化后(商品) |
|------|:--:|:--:|:--:|
| auto_merge ≥ | 0.80 | **0.53** | **0.54** |
| review ≥ | 0.50 | 0.50 | **0.51** |

> 🎯 **亮点 5**：系统能"越用越聪明"——每次新增审核日志后运行 `tune_thresholds.py`，阈值自动收敛。
