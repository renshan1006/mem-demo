# 调试报告

## 1. 已完成调试记录

### 1.1 数据生成阶段
- 问题：初始生成脚本未对 `record_id` 和 `canonical_customer_id` 做稳定分配。
- 解决：使用 `random.seed(42)` 和统一 `canonical_*` 映射，确保跨源重复一致性。

### 1.2 匹配管道阶段

- 问题：候选生成直接交叉全量数据导致计算量过大。
- 解决：加入 `region` 阻塞规则，只在相同地区源之间生成候选对。

- 问题：相似度打分无法区分"相似但不同实体"和"同一实体"。
- 解决（V1）：引入税号、电话精确匹配加权，并设定高置信度阈值。

- 问题：difflib 编辑距离无法处理"北京Alpha科技" vs "北京阿尔法科技"等语义变体。
- 解决（V2）：接入 bge-large-zh-v1.5 Embedding 语义匹配，名称相似度改用向量余弦相似度。客户审核真匹配率从 8.8% 提升至 39.9%。

- 问题：商品匹配中同名不同 SKU 的商品被误判为重复。
- 解决（V2）：对商品管道加入 SKU/UPC 冲突惩罚机制（上限 0.55），商品审核队列从 1,317 降至 80。

- 问题：匹配阈值硬编码为 0.80/0.50，无法自适应数据分布。
- 解决（V3）：实现 tune_thresholds.py 自动扫描 4,356 种阈值组合，结合人工审核日志微调，输出 optimized_thresholds.json 供管道自动读取。客户审核队列进一步降至 241，商品降至 31。

### 1.3 审核界面阶段

- 问题：Streamlit 页面初始无法显示数据。
- 解决：增加空队列判断，并提示先运行匹配管道生成审核文件。

- 问题：详情审核页面 HTML 被当作 Markdown 代码块显示为纯文本。
- 解决：引入 `textwrap.dedent` + `_md()` 包装函数消除 HTML 缩进。

- 问题：st.tabs() 不支持程序化切换，审核队列按钮无法跳转到详情页。
- 解决：替换为 st.radio 水平选择器 + session_state 控制。

- 问题：表单 key 不随记录变化，导航后表单值残留。
- 解决：表单及控件 key 绑定 current_index。

### 1.4 环境问题

- 问题：Python 3.9.25 + Windows SSL 证书存储损坏，Streamlit 启动时 tornado 崩溃。
- 解决：创建 run_streamlit.py 启动器，monkey-patch ssl.SSLContext.load_default_certs。

- 问题：HuggingFace 国内网络不可达，无法下载 bge-large-zh 模型。
- 解决：embedder.py 自动设置 HF_ENDPOINT=https://hf-mirror.com。

## 2. 目前已知风险点

- 审核日志数据量较小（客户 25 条，商品 6 条），阈值优化主要依赖真值标签。审核数据增多后应重新运行 tune_thresholds.py。
- Embedding 模型运行在 CPU 上，4200 条客户名称编码需 ~50 秒。数据量增大后可考虑 GPU 加速或向量缓存（Redis）。
- 当前所有中间数据仍为 CSV 文件存储，尚未接入 PostgreSQL/Redis。

### 1.5 Dify 集成阶段

- 问题：Dify DSL 手动编写的 YML 因版本格式不兼容导致导入失败（白屏/创建失败）。
- 解决：参照 Dify 0.6.0 真实导出格式，补齐 `type: custom` 包装层、时间戳节点 ID、`positionAbsolute`、`context.enabled` 等必需字段后重写 mdm_workflow.yml。

- 问题：Code 节点正则 `\{[^{}]*\}` 无法匹配嵌套 JSON（signals 子对象），解析走到了兜底默认值。
- 解决：改为括号计数法 `extract_json()`，从第一个 `{` 逐字符匹配到配对的 `}`，正确处理嵌套。

- 问题：Dify API 地址 `cloud.dify.ai` 404，Python 客户端调用不通。
- 解决：Dify Cloud 的实际 API 域名为 `api.dify.ai`，更新 `config.json` 和 `dify_client.py` 默认值。

### 1.6 数据库阶段

- 问题：客户和商品共用了同一个 ERP 数据源 ID 池（ERP-000001~ERP-001400），导致 `source_records` 表 UNIQUE(record_id, source) 约束冲突。
- 解决：将唯一约束扩展为 UNIQUE(record_id, source, entity_type)，区分客户和商品维度。

- 问题：migrate_to_db.py 中 `entity_type` 列未写入 `col_map`，导致 NOT NULL 约束失败。
- 解决：在列映射中补全 `entity_type` 和 `canonical_id` 字段。

### 1.7 数据质量评估智能体（M2）阶段

- 问题：Dify 一个 app 只能绑定一个已发布工作流，数据质量评估的 LLM 增强无法复用实体匹配工作流的 `api_key`。
- 解决：新建独立工作流 `dify/quality_workflow.yml`（开始→LLM 质量评估→解析 JSON→结束），配置独立 `quality_api_key`；`quality_agent.py` 自包含 `_call_quality_llm()` 读取 `DIFY_QUALITY_API_KEY` / `dify/config.json`，未配置时自动降级为纯统计报告，不报错（不动组长的 `dify_client.py`）。

- 问题：商品价格 ±5–8% 的注入变异用 IQR 异常值检测几乎检不出（IQR 对该幅度稳健，多数落界内）。
- 解决：价格差异改由**跨源一致性检测**呈现——按 `canonical_product_id` 分组，组内 `max/min > 1.05` 判为冲突（实测命中 58 组）；IQR 异常值检测器仅报告真实统计尾部，并在报告「评分方法说明」中注明方法学，保证答辩可辩护。

- 问题：Windows 控制台默认 GBK 编码，`python -c "..."` 打印中文/特殊符号（如 ✓）时抛 `UnicodeEncodeError`。
- 解决：验证脚本避免在 `-c` 内输出非 ASCII 字符；实际报告文件均以 UTF-8 写入磁盘，不受控制台编码影响。

## 2. 目前已知风险点

- 审核日志数据量较小（客户 25 条，商品 6 条），阈值优化主要依赖真值标签。建议在实际使用中积累更多审核数据后重新运行 tune_thresholds.py，阈值将逐步收敛。
- Embedding 模型运行在 CPU 上，4200 条客户名称编码需 ~50 秒。数据量增大后可考虑 GPU 加速或启用 Redis 向量缓存（cache_manager.py 已预留 Redis 接口）。
- bge-large-zh 模型文件约 1.3 GB，首次下载需要稳定网络。embedder.py 已配置 hf-mirror.com 镜像，但镜像也有间歇不可达的情况。可考虑将模型文件随项目分发或使用更轻量的 bge-small-zh（384 维，约 100 MB）。

## 3. 后续调试建议

- 增加审核数据量后重新运行 tune_thresholds.py，观察阈值在更多反馈下的收敛行为。
- 在真实数据（非合成）上测试匹配管道，验证跨域泛化能力。
- 制作答辩 PPT 与系统演示视频。
